## ▸ Architecture & Engineering Principles

<details>
<summary><strong>Expand — Engineering Philosophy</strong></summary>

<br/>

**Layered Architecture**

```
┌─────────────────────────────────────────────┐
│                 Presentation                │  ← Compose UI, ViewModels, UI State
├─────────────────────────────────────────────┤
│                   Domain                   │  ← Use Cases, Business Logic, Entities
├─────────────────────────────────────────────┤
│                    Data                    │  ← Repositories, Local DB, Remote API
└─────────────────────────────────────────────┘
              Dependency Rule: inward only
```

**Core Principles**

| Principle | Application |
|-----------|-------------|
| Single Responsibility | Every class does one thing. Every function has one reason to change. |
| Dependency Inversion | Business logic depends on abstractions, never on frameworks. |
| Separation of Concerns | UI knows nothing about data sources. Data knows nothing about UI. |
| Fail Fast | Validate at boundaries. Surface errors early, not deep in the stack. |
| Immutability by Default | State flows in one direction. Mutations are explicit and controlled. |

**Development Constraints**

- No business logic in ViewModels (it belongs in Use Cases)
- No coroutine dispatch in UI layer (ViewModels own scope, UI observes)
- No magic strings — constants, sealed classes, or enums
- No untested repository layer — data contracts must be verifiable

</details>

## ▸ Development Workflow

<details>
<summary><strong>Expand — How I work</strong></summary>

<br/>

**Project Initialization**
```
1. Define problem and user value — not features
2. Sketch data model and domain entities
3. Define repository contracts (interfaces first)
4. Scaffold layer structure before writing any logic
5. Wire dependency graph (Hilt modules)
6. Implement use cases against repository interfaces
7. Build ViewModel state machine
8. Compose UI against ViewModel state
9. Write tests at each layer boundary
10. Profile before optimizing
```

**Branching Strategy**
```
main          ← production-ready, tagged releases
develop       ← integration branch
feature/*     ← isolated feature work
fix/*         ← bug isolation
```

**Commit Convention**
```
feat(scope): description
fix(scope): description
refactor(scope): description
docs(scope): description
test(scope): description
chore(scope): description
```

</details>
